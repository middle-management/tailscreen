# Bidirectional Voice — Design

Status: draft
Date: 2026-04-28

## Goal

Add bidirectional voice between the sharer and all connected viewers during an active screen share. Multiple viewers can speak simultaneously and hear each other through the sharer (SFU-style relay). Voice rides on the existing port-7447 UDP socket.

## Scope

In:
- Voice on/off only while a screen-share session is active.
- Both ends muted by default; explicit toolbar/menu toggle to unmute.
- N-way audio: sharer plus any number of viewers, each can speak, each hears all others.
- Hardware AEC via `kAudioUnitSubType_VoiceProcessingIO`.

Out (v1):
- Voice without an active share.
- Push-to-talk hotkey.
- Per-viewer volume / mute controls.
- Noise suppression beyond VPIO defaults.
- Recording.
- Bandwidth adaptation, DTX (silence suppression).

## Constraints

- Must reuse the existing UDP/7447 socket. No new firewall hole.
- Must not regress video latency or reliability.
- Must honor macOS microphone permission (`NSMicrophoneUsageDescription`).
- Swift 6 strict concurrency: voice path is `Sendable`-correct, UI state stays on `@MainActor`.

## Architecture

```
Sharer process:
  AVAudioEngine(VPIO) ─mic─▶ AAC enc ─▶ RTP(PT=98, SSRC=0) ─▶ UDP fan-out to all viewers
                                                              ▲
  UDP recv ─▶ demux PT=98 ─▶ split:
                              ├─▶ relay to all viewers != sender (SSRC preserved)
                              └─▶ AAC dec per SSRC ─▶ AVAudioMixerNode ─▶ VPIO speaker

Viewer process:
  AVAudioEngine(VPIO) ─mic─▶ AAC enc ─▶ RTP(PT=98, SSRC=mine) ─▶ UDP to sharer
  UDP recv ─▶ demux PT=98 by SSRC ─▶ AAC dec per SSRC ─▶ AVAudioMixerNode ─▶ VPIO speaker
```

Sharer is the SFU: forwards each viewer's audio RTP packet to all other viewers without transcoding, and decodes locally so the sharer hears them. Each receiver runs one decoder per remote SSRC and mixes locally. SSRCs are issued by the sharer in a new `HELLO_ACK` control message.

## Wire format

Reuse RTP framing already established in `RTPPacket.swift`. Existing demux on the first byte (control 0x00–0x7F vs RTP V=2 0x80–0xBF) is unchanged.

New constants:

| Constant | Value |
|----------|-------|
| `RTPHeader.aacPayloadType` | `98` |
| `RTPHeader.audioClockHz` | `48_000` |
| `ScreenShareControlMessage.helloAck` | `0x04` |

Inbound RTP routing on receivers:

| PT | Codec | Clock |
|----|-------|-------|
| 96 | H.264 | 90 kHz |
| 97 | HEVC | 90 kHz |
| 98 | AAC-LC | 48 kHz |

AAC RTP payload framing: one AAC AU per RTP packet, no AU-headers. AU size = 1024 samples mono @ 48 kHz ≈ 21 ms. Target bitrate ≈ 32 kbps. Well under MTU.

`HELLO_ACK` payload: `[0x04][SSRC: 4 bytes BE]`. Sent by sharer on receipt of HELLO. Viewer waits up to 200 ms; on timeout retries HELLO once, then every 5 s. Without an SSRC the viewer never sends audio (video continues normally).

Sharer SSRC is fixed at `0x00000000`. Viewer SSRCs are random non-zero 32-bit values; sharer guarantees uniqueness within the session.

## Data flow

### Sharer outbound (mic → viewers)

1. `VoiceChannel` taps the VPIO input node at 48 kHz mono Float32.
2. Buffers accumulate to 1024-sample frames (~21 ms).
3. `AACCodec.encode` produces an AU.
4. `RTPAudio.pack(au, ssrc: 0)` wraps with PT=98 RTP header.
5. `TailscaleScreenShareServer` sends the packet to each registered viewer's UDP address.

### Viewer outbound (mic → sharer)

1. Same capture / encode pipeline as sharer.
2. `RTPAudio.pack(au, ssrc: assignedSSRC)`.
3. `TailscaleScreenShareClient` sends to the sharer's UDP address on the same socket the video uses.

### Sharer inbound (relay + local playback)

1. UDP recv loop sees an RTP packet with PT=98.
2. Look up viewer's address → SSRC. If unknown, drop.
3. For each *other* registered viewer, send the packet verbatim (preserving the sender's SSRC).
4. Hand the packet to local `VoiceChannel.receive`, which decodes per-SSRC and feeds the mixer.

### Viewer inbound (local playback)

1. UDP recv loop sees PT=98 RTP packet.
2. Look up decoder by SSRC; create one on first sight.
3. Decode → schedule on `AVAudioMixerNode` input bus for that SSRC.
4. Mixer feeds the VPIO output node so AEC reference signal is correct.

## Lifecycle

- App launch: no mic, no engine. No permission prompt.
- Share starts (sharer) or `Connect` succeeds (viewer): `AppState` instantiates `VoiceChannel` in muted state. Engine not started.
- First `toggleMic()` → on:
  - Request mic permission (lazy).
  - On grant: start `AVAudioEngine`, install input tap, begin sending RTP.
  - On deny: alert linking to Settings → Privacy → Microphone, leave button in `mic.slash`, `isMicOn = false`.
- `toggleMic()` → off: stop input tap, stop sending RTP. Receive/playback stays running so the user still hears others.
- Share disconnect: full `VoiceChannel` teardown. Sharer clears SSRC registry.

## Concurrency

- `VoiceChannel`: own internal serial queue for socket↔codec handoff. `@unchecked Sendable` like the existing networking classes; documents which methods are queue-confined.
- `AACCodec`: per-instance, not `Sendable`; only touched from the voice queue.
- `AppState` voice methods: `@MainActor`. They post work to `VoiceChannel` via async-unsafe sends; UI-bound state (`isMicOn`) stays on the main actor.
- No `Task { … self … }` from `deinit` (matches existing rule).

## Error handling

| Failure | Behavior |
|---------|----------|
| Encoder/decoder init fails | Log; `appState.showAlertMessage`; `isMicOn = false`. |
| Mic permission denied | Alert with Settings deeplink; toolbar stays `mic.slash`. |
| `HELLO_ACK` timeout | Retry HELLO at 200 ms, then every 5 s. Video unaffected. Voice silent until ack arrives. |
| RTP from unregistered SSRC on sharer | Drop. (Defends against spoofing within tailnet.) |
| RTP packet loss | Lost. No retransmit. Mixer underrun is silent. |
| Engine config change (device unplug) | Tear down and rebuild engine; remain muted afterwards. |

## Testing

Automated:
- `RTPAudioTests`: pack/unpack roundtrip, sequence wraparound at 0xFFFF, timestamp increments by 1024 per AU.
- `AACCodecTests`: encode then decode 1 s of 440 Hz sine, assert peak frequency within 1% and RMS within 3 dB.
- `VoiceChannelTests`: synthetic PCM through full enc → RTP → dec stack with mock send/recv closures; same waveform-tolerance check.
- E2E (gated on `TAILSCREEN_TS_AUTHKEY`, runs in `make test-e2e`): bring up two ephemeral nodes, sharer + viewer, send 1 s of synthetic audio from viewer, assert sharer receives ≥80% of frames decoded.

Manual (documented in README):
- AEC quality (laptop speakers vs headphones).
- 3-viewer SFU relay: every participant hears every other.
- Mic permission deny path.
- Device unplug / Bluetooth handoff mid-call.

## Migration / compatibility

Older clients (no audio support) are unaffected — they neither send PT=98 nor process `HELLO_ACK` (which they treat as an unknown control byte and drop). Sharer must therefore not assume HELLO_ACK is acknowledged; absence of audio from a viewer is normal. Older sharers will simply ignore incoming PT=98 RTP from a newer viewer (no relay path), which is the correct degradation — no error.

## Open questions

None blocking implementation. Bitrate (32 kbps) and frame size (1024 samples) chosen to match common voice profiles; tunable via constants in `AACCodec.swift`.
